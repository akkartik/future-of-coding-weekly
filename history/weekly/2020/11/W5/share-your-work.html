Martin Sosic üï∞Ô∏è 2020-11-17 16:27:09

Hi all,

for those who don't yet know, brother and I are working on https://wasp-lang.dev - a DSL (domain specific language) for building web apps with as little boilerplate as possible. It is open-source and integrates with React/Node.js/Prisma.

Main hypothesis of the experiment: What if web framework was a DSL (we are not the first ones to try this!)?

About half a year ago I posted a prototype here, and now we have released an Alpha! It is still very early, but we have docs, it can be properly installed, and you can build an app (though it still misses many important features).

Goal of the project and our focus is to make developing web apps as easy as possible, while still being flexible enough to be used for real projects in production. In that sense, I think it should be an ok fit for FoC, since it should allow developers who are not very experienced in web development to build a full-stack production-ready web app by themselves. It does add one more layer of abstraction, but the goal is to reduce knowledge barrier.

Bigger, crazier vision is to one day also build a no-code visual editor on top of Wasp, which would allow non-developers to participate alongside developers. We are thinking this could work very good with Wasp since it is declarative, and if we manage to capture big portion of web app functionality in this declarative code, it could be easier to build such editor on top.

When using Wasp, you write code in Wasp (declarative) and Javascript together.

Compiler implemented in Haskell compiles/transpiles the source code of Wasp project into a human-readable JS based web app (which you don't care about unless you decide to eject -> take the code and leave Wasp).

We started working on Wasp about a year ago, first as a side project and then full time (we are bootstraping), and we are excited to see how far we can get with the whole concept! Right now we are trying to get first testers and contributors (actually got our first contributor couple of days ago wohooo!), while we also keep developing Wasp.

Webpage: https://wasp-lang.dev

Docs: https://wasp-lang.dev/docs/

Todo App tutorial: https://wasp-lang.dev/docs/tutorials/todo-app <- probably best place to get started and try it out

Github repo: https://github.com/wasp-lang/wasp

Thanks for any feedback!

Konrad Hinsen 2020-11-25 09:58:24

I agree with most of what you say, but keep wondering of what exactly you mean by "code". Most often that term is used for algorithms and data structures expressed in a programming language. However, many scientific ideas are not of that kind. The most powerful scientific models we have are expressed as specifications, not algorithms, and programming languages are not a good medium to express them.

Shalabh Chaturvedi just shared a paper by Terry Winograd on Twitter (https://twitter.com/chatur_shalabh/status/1331410504204488706) that addresses this point very well (and is 40 years old!). It describes "Three domains of descriptions" of software systems. Scientific concepts, models, and ideas belong to the "subject domain", whereas code is designed for the "domain of implementation".

However, if you define "code" in a wider sense that includes formal languages designed for other purposes than "programming", then I agree with everything you say üòâ
