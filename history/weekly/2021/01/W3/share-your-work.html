David Piepgrass 2021-01-15 14:04:00

This is one of the many things text editors don't do that seem obvious to me. Others include:

  • Elastic tabstops (makes proportional fonts more usable in code)
  • respecting the local whitespace style when inserting newlines (i.e. if I press enter and the previous line starts with 2 tabs followed by 8 spaces, the new line should start with 2 tabs followed by 8 spaces)
  • highlighting definitions differently than usages, e.g. the "x" in "int x" might have an outline around it while the "x" in "x++" would not. As another example, there should be a horizontal line across the editor above every top-level function and class, so that it is easier to see where each new function/class begins. Also, when scrolling down into a large class/function, keep the class/function header visible on the top line of the editor so I can always see at a glance what context I am in
  • using colors primarily to categorize instead of just representing syntax - e.g. different colors for standard library functions, functions in our Backend API, and functions in the Frontend API. Scratch figured this one out. Multiple categorizations I can switch between with hotkeys might be useful.
  • Allowing emojis/symbols to be attached to specific entities of interest so it is easier to see at a glance when one of those entities is being used, e.g. attach ‚ù¶ to Console.WriteLine so it "jumps out" at me more. Alternately, could use symbols to abbreviate long-winded code, e.g. hiding Console.WriteLine and showing ‚ù¶ in its place.
  • Defining language keywords that attach to statements and cause them to be highlighted/enlarged or dimmed/shrunk in the editor. Often certain "key" parts of a function or class are worth highlighting, and other parts that are pure incidental complexity (e.g. writing a log message or verifying a security flag) and there should be a way to emphasize one and de-emphasize the other.

Breck Yunits 2021-01-17 00:44:43

Thanks for the feedback Konrad!

I would say at this stage of the game the 2-D/3-D aspect is

important for the UI of programming. It's like WYSIWYG of

programming.

In Tree, scope has a semantic meaning but also a geometric

meaning‚Äîif things are down and to the right, they belong

together.

In Tree, there aren't infinite ways to write a list. There's

just one, and it's the way you would do it in the real

world: put items next to each other.

The languages follow constraints imposed by geometry. While

this may seem like it would limit creativity, in fact I've

found the opposite. By letting you focus on signal, and

removing all noise, you can focus on the semantics, and

really simplify your code to its essence.

There are also lots more you can do when you break out of

1-dimensional, linearly parsed languages. Tree Notation is a

subset of Grid Notation. There are other 2-D notations

out there.

An example might help. Would be easier to do a visual

but could try to put it into words. Imagine a project

wouldn't be a folder with a bunch of files, but instead one

file that you would open with a spreadsheet like IDE. and

edit like a canvas. What you'd normally put into files you

would instead drop an "anchor" in some cell in the sheet,

and start typing. That would be a root tree. And you could

have many of them throughout the canvas, instead of 1 file.

This is a good structure for languages for simulation. There

are some things like this, but none of them (AFAIK) have

ever made the layout and code isomorphic, and when you do

that you don't give up anything, and you make parsing and

tool building a lot easier.
