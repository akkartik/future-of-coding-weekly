I've been trying to convince people that we should standardize the concept of a "generic expression language"... so I designed this: http://loyc.net/les/

David Piepgrass 2021-02-01 02:43:06

It occurred to me today that, by combining tree-sitter & Loyc, it should be straightforward to convert source code of various languages to a common tree format. For example, the Python code

Leading comment

if c > 0:

    x = f

else:

    y = f(123);

could be converted to an identical syntax tree as the one produced by the C/C#/Java code

// Leading comment

if (c > 0) {

  x = f;

} else {

  y = f(123);

}

Specifically, I could write a converter that would take any output from tree-sitter and convert it to a Loyc tree. I could then use LeMP macros to straightforwardly convert the syntax tree of a particular language into a mostly-common format. And comments could be preserved too, as I have written a language-agnostic algorithm for this purpose.

Now, from the readme and article about tree-hugger it doesn't look like tree-hugger does something like this, but correct me if I'm wrong.

This wasn't the problem I was interested in solving - I wanted to figure out how to go in the reverse direction, from Loyc tree to text, because part of my dream is to help people convert code between languages. But seeing this video makes me want to ask if conversion to a common format is something that would help the Docly project. If so, I available to help at below-market rates.

You could certainly have a separate ML model for each language - there may be enough code on GitHib to support this approach, at least for popular languages, but perhaps not for new or obscure languages. But if you convert all languages to a common format, a single model might be able to share intuitions across languages, potentially allowing improved summarization in a single model.

Adnan Chaumette 2021-02-05 16:40:53

Recently shared this demo of GraphN (VPL made for 3D artists in the computer graphics industry) but forgot to share it here as well üòÖ

https://www.youtube.com/watch?v=_FscRL045z4

This was a nice fun project that aimed to solve a problem that has been requested from the Autodesk Maya team for decades (not kidding): how do you drag an object on any surface, and have it position and align itself to the normal of the polygonal face under your cursor.

The graph that you're seeing basically queries the mouse's screen space position, converts it to world space position, then shoots a ray from the camera to the world to detect the hit position and the orientation/normal of the position that was hit. This position + orientation is then used to set the position/rotation of the object pTorus1 in the scene.

The artist of course doesn't see any of this: they just have basic nodes that allow them to query such data, every node and its properties are rich with descriptions, making it easy to hover over anything and know a) what it does and b) what it expects as a connection.

There's still lots of things that I'd love to improve on the user experience here. If you ask me how satisfied I am with the UX I'd say 30%, because like with any VPL there's always this impression that you're missing something so obvious yet crucial to really changing the game üòÑ
